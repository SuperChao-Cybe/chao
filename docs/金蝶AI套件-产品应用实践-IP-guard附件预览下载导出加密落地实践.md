# 金蝶AI套件产品应用实践：IP-guard 附件预览、下载、导出加密落地

## 一、项目背景

在金蝶 AI 星空旗舰版项目中，附件安全通常不是一个“可有可无”的小需求，而是很多客户上线前必须明确落实的控制项。

尤其是以下几类场景，客户会非常关注：

- 外网访问 ERP 时，附件预览是否需要加密
- 外网下载附件时，文件是否必须经过加密保护
- 单据导出、附件导出时，是否也要纳入统一的安全控制
- 内网办公场景下，是否还能保持正常预览和使用体验
- 后续如果公网地址、节点、Jar、配置变化，实施和运维是否还能快速维护

如果这些问题处理不好，现场通常会出现两类典型风险：

1. 安全要求达不到，外网预览、下载、导出仍然是明文文件链路。
2. 加密策略“控死了”，导致内网预览乱码、未保存附件无法预览、生产发布后旧 Pod 残留、版本不一致等问题。

所以，这类需求的难点并不只是“把加密功能做出来”，而是要真正做到：

- 外网链路可控
- 内网链路可用
- 运维发布可落地
- 后续维护可持续

## 二、项目目标

这次落地的目标，围绕金蝶 AI 星空旗舰版附件链路，主要分成三部分：

### 1. 业务安全目标

- 实现外网附件预览加密
- 实现外网附件下载加密
- 实现附件或单据导出加密

### 2. 使用体验目标

- 内网正常访问时，尽量保持原有预览体验
- 未保存附件、临时文件链路尽量兼容实际业务操作
- 避免因为加密策略过于死板，影响日常办公

### 3. 交付维护目标

- 方案能发布到生产环境
- 运行时开关可控
- 版本、Jar、MD5、当前状态可核验
- 客户后续能通过维护工具自行完成常规操作，而不是长期依赖命令行

## 三、方案设计思路

这次实践的核心，不是单点改一个下载接口，而是按金蝶 AI 星空旗舰版的实际链路，把附件安全分成三条线处理：

### 1. 预览链路

对于外网预览，按照配置决定是否走 IP-guard 加密链路。  
同时对内网场景做区分，避免把内网预览也一并加密，导致用户看到乱码或无法直接打开。

### 2. 下载链路

下载链路重点解决两个问题：

- 下载动作本身是否已经经过加密
- 下载后的文件名、临时文件名、后缀名是否与加密后的实际文件一致

这部分如果只做加密、不处理临时文件和响应头，最终用户侧往往会出现“日志显示加密成功，但下载下来的文件看起来还是原文件”的假象。

### 3. 导出链路

很多项目只处理了附件预览和下载，没有把导出链路纳入统一控制。  
但在客户实际使用中，导出往往也是敏感数据外发的重要通道，所以导出加密必须和附件下载一起考虑，而不能遗漏。

## 四、落地过程中遇到的典型问题

这类方案在现场实施时，真正耗时间的往往不是功能编码，而是链路辨识和部署一致性。

本次实践里，比较典型的问题包括：

### 1. 预览加密了，但下载没有加密

从日志看已经进入了加密逻辑，但用户下载结果仍然不符合预期。  
这时不能只盯代码逻辑，还要继续核对：

- 入口 URL 到底走的是哪条链路
- 是否经过前端转发、临时文件下载地址或其他网关链路
- 返回的是否真的是加密后文件内容
- 客户端看到的文件名和后缀是否已经被正确改写

### 2. 内网预览也被加密，结果直接乱码

这类问题说明策略判断不够精细，把“外网安全要求”直接覆盖到了“内网办公体验”上。  
如果不做区分，用户会觉得系统“功能坏了”，而不是“安全增强了”。

### 3. Jar 已替换，但 Pod 里仍混有旧版本

这在集群环境中非常常见。  
表面上看 deployment 已 rollout 成功，但实际 Pod 中可能仍然混有旧 Jar，导致：

- 有的节点生效
- 有的节点不生效
- 同一个功能不同时间点表现不一致

所以这类项目不能只看 rollout status，还必须核对 Pod 中实际 Jar 的 MD5。

### 4. 功能能跑，但维护成本太高

如果后续每次改地址、改节点、改 Jar、改开关，都要让客户去服务器上敲命令，这套方案就不是真正可交付。  
生产方案最终一定要落实到“客户能维护、运维能复盘、实施能交接”。

## 五、最终落地效果

经过调整后，这套方案最终达成的效果主要有：

### 1. 外网文件链路纳入统一安全控制

- 外网附件预览可按策略加密
- 外网附件下载可按策略加密
- 导出链路可按策略加密

### 2. 内外网访问逻辑分离

不是简单“一刀切”全部加密，而是按访问场景和链路做区分。  
这样既满足安全要求，也尽量保留内网正常办公体验。

### 3. 发布和核验更标准化

除了代码和 Jar，本次还同步整理了：

- 版本记录
- 发布包说明
- GitHub 分发结构
- 维护工具说明
- 旧 Pod 清理和状态核验思路

### 4. 智能体可复用

这套成果不仅仅是一次项目交付，也被整理成了一个可复用的智能体技能/知识包。  
后续再遇到类似的金蝶 AI 星空旗舰版附件安全需求，可以直接复用这套结构，而不是从零开始重新分析。

## 六、为什么还要整理成可分发仓库

在项目后期，我把这套内容进一步整理成了 GitHub 仓库，原因很直接：

- 方便实施、运维、开发统一查看资料
- 方便智能体复用和适配
- 方便后续版本演进和经验沉淀
- 方便客户侧在需要时快速回看方案说明、发布说明和使用说明

这里要特别说明一点：

这个仓库虽然支持 Codex marketplace / plugin 方式安装，但它并不是“只能给 Codex 用”。  
更准确地说，它是一个围绕 **金蝶 AI 星空旗舰版 IP-guard 加密方案** 整理出来的可交付知识包：

- Codex 可以直接安装使用
- 其他智能体可以读取其中的技能和参考文档做适配
- 实施、运维、开发也可以直接把它当成方案资料库使用

## 七、对类似项目的几点建议

如果后面还会做金蝶 AI 星空旗舰版文件安全类需求，我建议优先注意以下几点：

### 1. 不要只盯“功能是否写完”

附件预览、下载、导出，实际上是三条不同链路。  
只处理其中一条，现场大概率还会返工。

### 2. 一定要区分内外网策略

安全要求不能简单覆盖所有链路。  
否则最容易出现“外网安全做到了，内网办公反而不能用了”的问题。

### 3. 集群环境必须核对实际 Jar 一致性

生产问题很多不是“代码没改对”，而是“有的 Pod 还是旧版本”。  
如果没有版本和 MD5 核验习惯，排障会非常低效。

### 4. 方案必须考虑客户后续维护

真正能长期运行的方案，不只是上线那天能成功，而是后面改配置、改地址、发新 Jar、查状态时仍然能顺利维护。

## 八、结语

这次实践本质上做的，不只是一个“附件加密功能”，而是把金蝶 AI 星空旗舰版中一条容易出问题、又非常贴近客户安全诉求的文件链路，整理成了一套更完整的产品应用实践。

如果后续还会继续做类似项目，我认为最有价值的沉淀并不是某一段代码，而是：

- 哪些链路必须分开看
- 哪些部署问题一定会反复出现
- 哪些维护能力必须提前交付给客户

把这些经验整理出来，后面的项目推进效率会明显高很多。
